# 02. トランスポートとフレーミング

新 API は **1 つのメッセージ層**（`03-message-protocol.md`）を、複数の物理リンクの上に載せる。
リンク差分はこの層で吸収し、上位のメッセージ意味論は完全に共通とする。

```
+-----------------------------------------------------------+
|  メッセージ層 (RDP)  : 要求/応答 + イベント + テレメトリ   |
+-----------------------------------------------------------+
|  フレーミング層      : COBS フレーム + CRC + シーケンス    |
+-------------------------+---------------------------------+
| ストリーム適応           | GATT 適応                        |
| USB CDC-ACM / BT SPP    | BLE (Notify/Write 特性)          |
+-------------------------+---------------------------------+
```

## 1. 物理リンクの種類

| リンク | 性質 | MTU/単位 | 用途 |
|--------|------|----------|------|
| **USB CDC-ACM**（仮想シリアル） | バイトストリーム | 実効 64B バルク（集約可） | PC、Android(USB) |
| **Bluetooth Classic SPP** | バイトストリーム | リンク依存（数百 B） | 旧 HC-05 系モジュール |
| **BLE GATT** | パケット（特性 R/W/Notify） | 交渉後 23〜247B | スマホ標準、低消費電力 |

STM32F4/F7 自体は BLE を内蔵しないため、外付け BLE モジュール（nRF52 / ESP32 等）または
SPP モジュールを UART 経由で接続する構成を想定する。**ファーム側から見ればいずれも
UART/USB のバイトストリームか、GATT ブリッジのどちらか**になる。

## 2. ストリーム適応（USB CDC / BT SPP）

バイトストリームはメッセージ境界を持たないため、**COBS（Consistent Overhead Byte Stuffing）**で
フレーム化する。COBS はフレーム区切りバイト `0x00` を本文中から除去するため、
**自己同期**（途中接続でも次の `0x00` から復帰）でき、エスケープ処理が単純で組込み向き。

### フレーム（COBS デコード後の中身）

```
+--------+--------+--------+--------+-----------+--------+
| VER    | FLAGS  | SEQ    | LEN    | PAYLOAD   | CRC16  |
| u8     | u8     | u16 LE | u16 LE | LEN bytes | u16 LE |
+--------+--------+--------+--------+-----------+--------+
        ↑ ここ全体を COBS エンコードし、末尾に区切り 0x00 を付与して送出
```

- `VER` : フレーミング層バージョン（初期値 `0x01`）。
- `FLAGS` : bit0=応答要求, bit1=断片化あり, bit2=最終断片, 他予約。
- `SEQ` : 要求/応答相関とテレメトリ連番に使う 16bit シーケンス。
- `LEN` : `PAYLOAD` のバイト長。
- `CRC16` : `VER..PAYLOAD` に対する CRC-16/CCITT。`PAYLOAD` は CRC 計算に含む、`CRC16` 自身は除く。
- フレーム最大長は能力ネゴシエーション（`Hello.max_payload`）で決まる。
  既定は 256B、拡張で最大 4096B。

> CRC は CRC-16/CCITT（多項式 0x1021）を採用。旧プロトコルの CRC32 から変更し、
> 小 MTU での相対オーバーヘッドを削減する。フレーミングは `device-api` クレートで共有実装する。

## 3. GATT 適応（BLE）

BLE では COBS は使わず、GATT 特性をフレーム境界として用いる。
1 つのプライマリサービス `RustEMS Service` の下に次の特性を定義する。

| 特性 | プロパティ | 役割 |
|------|-----------|------|
| **Control Point** | Write / Write-No-Response | ホスト→デバイスの要求フレーム |
| **Response** | Notify | デバイス→ホストの応答フレーム |
| **Telemetry** | Notify | テレメトリ push（高頻度・別ストリーム） |
| **Event** | Notify | 非同期イベント push |
| **Info** | Read | `Hello` 相当の静的情報（接続直後の即読み用） |

- ペイロードが ATT_MTU を超える場合は**断片化**する。フレームヘッダの
  `FLAGS.bit1/bit2`（断片化あり/最終断片）と `SEQ` を使って受信側で再結合する。
- UUID は 128bit ベンダ固有を割り当てる（値は実装時に確定。本書では役割のみ規定）。
- テレメトリとイベントを応答とは**別の特性**に分けることで、設定 RPC と高頻度 push の
  多重化（要件 N6）を GATT レベルで自然に実現する。

## 4. 断片化（共通）

メッセージが 1 フレームに収まらない場合（大きなカタログ、テーブル一括取得、BLE 小 MTU）：

- 送信側は `SEQ` を共有したまま複数フレームに分割し、各フレームに
  `bit1=断片化あり` を立て、最後のフレームのみ `bit2=最終断片` を立てる。
- 受信側は `SEQ` ごとに再結合バッファ（`heapless::Vec`、上限は能力で公開）に積む。
- 上限超過・タイムアウト時は `Error{ code = Fragmentation }` を返してバッファを破棄する。

## 5. 接続ライフサイクル

```
1. リンク確立（USB 列挙 / BT ペアリング / BLE 接続）
2. ホスト → System.Hello 要求      ──┐ 能力・プロトコル版・スキーマハッシュ取得
   デバイス → Hello 応答           ──┘
3. プロトコル版が非互換なら切断（メジャー不一致）
4. スキーマハッシュがキャッシュと不一致なら Descriptor.* で再取得
5. 通常運用（Config RPC / Telemetry 購読 / Event 受信）
6. アイドル時は Ping で死活監視（既定 2s 間隔、3 回欠落で切断扱い）
```

## 6. 実装メモ

- フレーミング・COBS・CRC・断片化は新クレート `device-api`（`no_std`, ヒープ無し）に置き、
  デバイス側 / ホスト側（`client`）で共有する。
- USB は `embassy-usb` の CDC-ACM、UART/SPP は `UartPort` トレイト、
  BLE は外部モジュールのブリッジ実装にそれぞれ適応層を用意する。
- ホスト側（PC/スマホ）は tokio ベースの非同期実装、または各 OS の BLE API でラップする。
